# UI スタイルガイドとは：定義と活用の指針

UI スタイルガイドは、デザインシステムの原理原則を具体的な UI 部品として具現化した**「見た目」の見本帳**である。本ドキュメントでは、各コンポーネントが備えるべき物理的な数値や振る舞いの詳細を規定し、実装者が迷わずにインターフェースを構築するための基準を定める。

---

## 1. 記述すべき必須要素

スタイルガイドは、単なるビジュアルの提示に留まらず、以下の具体的な「仕様データ」を網羅しなければならない。これにより、異なる開発環境であっても同一の視覚的結果を得ることが可能となる。

- **具体的な数値**: 余白（Margin/Padding）、コンポーネントのサイズ、タイポグラフィ（フォントサイズ、行間、太さ）。これらは原則としてデザインシステムで定めた基本単位（例：4px）の倍数で記述する
- **色彩設計（セマンティクス）**: 使用する色のコード（HEX/RGBA）と、その「用途（例：背景、境界、警告、成功）」の対応付け。デザイン・トークン名との紐付けを明確にする
- **状態（State）別の定義**: 通常時（Normal）、マウスホバー時（Hover）、クリック時（Active/Pressed）、フォーカス時（Focus）、および無効化時（Disabled）における、色・形状・透過度の変化を詳細に規定する
- **インタラクションとアニメーション**: 変化の所要時間（ms）、およびイージング曲線（Easing Curve）。「パッと切り替わる」のか「滑らかに動く」のかを数値で定義する

---

## 2. プラットフォーム非依存の原則

本プロジェクトのスタイルガイドは、特定の UI フレームワーク（WPF, React, Flutter 等）に依存しない**「論理的なデザインスペック」**としての記述を基本とする。

#### 論理仕様と物理実装の分離

ドキュメントの主文では「背景色はアクセントカラーの透過 15% とする」といった論理的なルールを記述する。特定のフレームワーク向けの実装コード（例：XAML の Style 定義や CSS クラス）は、あくまでその論理仕様を具現化するための「リファレンス実装（例示）」として扱うものとする。これにより、異なるテクノロジースタックを採用する別プロジェクトへのデザイン展開を容易にする。

---

## 3. デザインシステムとの棲み分け

- **デザインシステム**: 「仕組み」の憲法。なぜその設計思想なのかという哲学、原理原則、および共通の変数を定義する（Why & How）
- **UI スタイルガイド**: 「見た目」の見本帳。デザインシステムに基づいた具体的な UI コンポーネントの論理仕様と実装例を提示する（What）

---

## 4. 構成ドキュメント

具体的なスタイル定義は、役割や画面ごとに以下のドキュメントへと展開する。

- **[ウィンドウ・レイアウト](window_style.md)**: 基本構造や共通のレイアウトパターン
- **[設定画面スタイル](settings_style.md)**: フォームやナビゲーションの具体的な構成見本
- **コントロール共通**: ボタンや入力フィールドなどの最小単位の部品定義（※プラットフォームごとの実装サンプルを併記することを推奨）
