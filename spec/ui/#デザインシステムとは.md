# デザインシステムとは：定義と運用の指針

デザインシステムは、単なる「見た目のカタログ」ではない。それは、ユーザーに対して**一貫した体験（UX）**を約束し、開発者に対して**迷いのない意思決定**を促すための「共有言語」である。本ドキュメントでは、本プロジェクトにおけるデザインシステムの定義とその運用ルールを規定する。

---

## 1. 導入の目的

本プロジェクトにおいてデザインシステムを構築・維持する目的は、以下の 3 つの価値を最大化することにある。

- **体験の一貫性**: アプリ全体のトーン＆マナーを統一し、プロフェッショナルな品質をユーザーに提供する
- **意思決定の高速化**: UI 作成時に、色、余白、インタラクションを再定義する手間を省き、本質的な機能実装に集中できる環境を作る
- **実装品質の担保**: バグの温床となる「色のハードコード」や「場当たり的なスタイル指定」を排除し、保守性を高める

---

## 2. 遵守すべき三原則

デザインに関わるすべての判断は、以下の三原則に照らして行われるものとする。

#### 一貫性 (Consistency)

同じ機能を持つ要素は、アプリのどこにあっても同じ見た目、同じ挙動をしなければならない。これによりユーザーの学習コストを最小化し、予測可能な操作感（心地よさ）を提供する。

#### 適応性 (Adaptability)

ライトテーマとダークテーマの両方において、同等の使いやすさと視覚的な快適さを提供しなければならない。単なる色の反転ではなく、それぞれの環境における視認性の最適化を追求する。

#### 保守性 (Maintainability)

すべてのデザイン要素は、XAML リソースによって一元管理されなければならない。個別の View で「色」や「数値」を決定することを禁止し、将来の変更が全画面に即座に、かつ安全に行き渡る構造を維持する。

---

## 3. 運用プロセスとルール

ドキュメントと実装の乖離を防ぐため、以下のプロセスを遵守する。

- **マスター・オブ・リソース**: 常に XAML リソース (`LightColors.xaml`, `DarkColors.xaml`) を最終的な正解（真実のソース）とする
- **スタイルの抽象化**: 個別のコントロールへの直接的なプロパティ記述を避け、意味に基づいた共通 `Style` の定義と適用を徹底する
- **変更のワークフロー**: 新しいデザインパターンが必要になった場合は、まず本ドキュメントおよび `design_system.md` への定義追加を行い、その後に実装へ着手する。実装が先行する「場当たり的な修正」は認められない。

---

## 4. UI スタイルガイドとの棲み分け

本ドキュメント（デザインシステム）と、別途定義される UI スタイルガイドは、以下の役割分担に基づいて運用する。

- **デザインシステム（本紙）**: 「仕組み」の憲法。なぜその色や余白なのかという哲学、原理原則、および共通のデザイン・トークンを定義する
- **UI スタイルガイド**: 「見た目」の見本帳。デザインシステムを具現化した具体的な UI 部品（`window_style.md`, `settings_style.md` 等）の実装見本を提示する
