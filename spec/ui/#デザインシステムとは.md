# デザインシステムとは

## 1. 目的

デザインシステムは、単なる「見た目の定義」ではなく、ユーザーに対して**一貫した体験（UX）**を提供し、開発者に対して**効率的で迷いのない実装指針**を与えるための「共有言語」である。

本プロジェクトでは、以下の 3 つを主な目的として運用する。

- **一貫性の保証**: アプリ全体のトーン＆マナーを統一し、プロフェッショナルな品質を維持する
- **意思決定の高速化**: UI 作成時の色、余白、インタラクションの挙動に関する再定義を不要とする
- **実装品質の担保**: バグの温床となる「色のハードコード」や「場当たり的なスタイル指定」を排除する

## 2. 三原則

デザインシステムの運用にあたり、以下の三原則を遵守するものとする。

#### 一貫性 (Consistency)

同じ機能を持つ要素は、アプリのどこにあっても同じ見た目、同じ挙動をしなければならない。これによりユーザーの学習コストを最小化し、予測可能な操作感を提供することを目的とする。

#### 適応性 (Adaptability)

ライトテーマとダークテーマの両方において、同等の使いやすさと視覚的な快適さを提供しなければならない。単なる色の反転ではなく、それぞれの環境における視認性の最適化を目指すものとする。

#### 保守性 (Maintainability)

すべてのデザイン要素は、XAML リソースによって一元管理されなければならない。個別の View で「色」や「数値」を決定することを禁止し、変更の影響範囲を明確にするものとする。

## 3. 運用ルール

- **マスター・オブ・リソース**: 常に XAML リソース (`LightColors.xaml`, `DarkColors.xaml`) を正解とする
- **スタイルの抽象化**: 個別のコントロールへの直接的なプロパティ記述を避け、共通 `Style` の定義と適用を徹底する
- **変更のプロセス**: 新しいデザインパターンが必要になった場合は、まずデザインシステムへの定義追加を行い、その後に実装へ着手するものとする
