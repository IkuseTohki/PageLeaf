# UIデザイン原則

PageLeaf における UI 設計の「大前提」となる 5 つの原則を定義します。本プロジェクトのすべての UI コンポーネントおよび画面構成は、この原則に基づいて設計・実装されなければなりません。

---

## 1. 一貫性（Consistency）

**「同じ意味には同じ表現を、異なる意味には異なる表現を」**

ユーザーが一度学習した操作が、他の画面でも通用することを保証します。

- **視覚的ボキャブラリの統一**: 同じ機能を持つボタン（例：保存、閉じる）は、全画面で共通の色、アイコン、配置ルールを適用します。
- **強調色の限定**: 強調色（Accent Color）は、その画面でもっとも重要な 1 箇所のインタラクションのみに使用し、視線を迷わせないようにします。
- **配置の定石化**: ダイアログの「OK」ボタンは常に右側、「キャンセル」は左側といった、OS やプロジェクト内の慣習を守ります。

## 2. 予測可能性（Predictability）

**「次に何が起こるか、ユーザーが確信を持てる状態にする」**

意図しない挙動（サプライズ）は、ユーザーの集中力を削ぐ最大の要因です。

- **フィードバックの即時性**: クリック時や処理実行時には、ホバー効果、アニメーション、または状態インジケータ（Dirty ドットなど）を通じて、システムが反応していることを示します。
- **標準への準拠**: 独自性の高い特殊な UI よりも、Windows/WPF の標準的な挙動や、一般的なコードエディタの慣習を優先します。
- **非破壊的な操作**: 取り返しのつかない操作（削除、保存せず閉じる）の前には、必ず確認を挟むか、状態を明示します。

## 3. 近接とグルーピング（Grouping & Proximity）

**「関係の深さを物理的な距離で表現する」**

視線の移動を最小限にし、情報の構造を一目で理解できるようにします。

- **関連要素の集約**: たとえば「フォントサイズ」と「フォントファミリー」のように、相互に関連する設定項目は同じグループ（Border や StackPanel 内）に配置します。
- **余白による構造化**: グループ間には適切な余白（Margin）を与え、線（Separator）に頼らなくても情報の区切りが認識できるようにします。
- **コンテキストの維持**: 編集対象のすぐそばに操作 UI を置く（例：プレビュー上の要素をクリックして CSS パネルへ誘導）など、作業対象とツールの距離を縮めます。

## 4. 省略より明確さ（Clarity over Cleverness）

**「短い文言より、迷わない文言を」**

UI の美しさを優先して言葉を削り、ユーザーを困惑させてはいけません。

- **自己説明的なラベル**: アイコンのみのボタンは避け、可能な限りツールチップ（ToolTip）を付与するか、ラベルを添えます。
- **具体的なメッセージ**:
  - ❌ 「確認」
  - ✔️ 「未保存の変更を保存して終了しますか？」
- **エラーの回避**: ユーザーが間違った後に叱るのではなく、入力制限（数値のみ、負の値の抑止など）を用いて、間違いようのない仕組み（ポカヨケ）を構築します。

## 5. ゼロ状態の設計（Empty States & Guidance）

**「データがない時こそ、ユーザーを導くチャンスにする」**

何も表示されない「空白」を放置せず、次の一手を案内します。

- **オンボーディング**: 起動直後のファイルが開かれていない状態、検索結果がゼロの状態などに、操作のヒントを提示します。
- **ヒントの提示**:
  - ✔️ 「まだファイルが開かれていません。Ctrl+O でファイルを選択するか、ここにドラッグ＆ドロップしてください。」
- **プレースホルダーの活用**: フロントマター編集パネルなどで、未入力時に「タイトルを入力...」などの期待される形式を薄く表示します。
